# Introduction

Retrieval Augmented Generation (RAG) is a technique where context is loaded from a datastore and the model is directed to work on the data provided.

Based on concepts presented in https://python.langchain.com/docs/tutorials/rag/

Pipeline Components used
 - Prompt Template: based on https://smith.langchain.com/hub/rlm/rag-prompt
 - Embedding Model: OpenAI Embeddings (`text-embedding-3-small`), remote access (Requires paid account)
 - Large Language Model: OpenAI Chat, `gpt-5.4-nano`, remote access (Requires paid account)
 - Vectorstore: Chroma, local instance persisted to `chroma_db/` (Open source)

Paid model access should cost a few cents of usage fees.

The three pipelines share configuration (model names, logging, the vectorstore) through `rag_common.py`:

 - `index_pipeline.py` — load a web document, split it, embed it, store it in Chroma
 - `rag_pipeline.py` — retrieve context and answer a question
 - `eval_pipeline.py` — grade the answer three ways (see [Evaluation](#evaluation))

# Concepts

First we index the data and then we do RAG using the user's question.

## Indexing

Indexing involves making the data searchable. The process involves

1. Load: Load documents using a **Document Loader**
2. Split: Divide documents into chunks with a **Text Splitter**
3. Store: Convert text to numbers with an **Embedding Model** and store it in a **Vector Store**

 ![Indexing Pipeline](/images/indexing-pipeline.png?raw=true "Indexing Pipeline")

## Retrieval Augmented Generation

1. Retrieve: Use the user input to retrieve relevant context chunks using a **Retriever**
2. Generate: A **Chat Model** or **LLM** generates a response using the prompt and context

 ![RAG Pipeline](/images/rag-pipeline.png?raw=true "RAG Pipeline")

# Dependencies

All of the dependencies listed below need to be in place before running the code.

 - Funded OpenAI account
 - Python virtual environment
 - Environment variables

ChromaDB ships as the `langchain-chroma` dependency in `requirements.txt`; it runs in-process and writes its files to `chroma_db/` (gitignored, regenerated by the index pipeline). No separate server is required.

## OpenAI Account

You'll need an OpenAI developer account and an API key. You will also need to purchase credits before making calls to the embedding and chat models.

## Python Virtual Environment

Set up a Python virtual environment ([instructions](../../docs/setup/python-venv.md)), then install this example's dependencies with `pip install -r requirements.txt`.

## Environment Variables

Copy `.env.example` to `.env` and fill in your key:

```
cp .env.example .env
```

| Variable | Required | Purpose |
| --- | --- | --- |
| `OPENAI_API_KEY` | yes | OpenAI key from a funded account. |
| `LOG_LEVEL` | no | Logging verbosity (`DEBUG`/`INFO`/`WARNING`/`ERROR`). Default `INFO`. |
| `SOURCE_URL` | no | Document indexed by `index_pipeline.py`. |
| `QUERY` | no | Question asked by `rag_pipeline.py` / `eval_pipeline.py`. |
| `SKIP_HUMAN_EVAL` | no | Set to any value to skip the interactive human-grade step. |

Running any pipeline without `OPENAI_API_KEY` set prints a single friendly line (`Missing OPENAI_API_KEY ...`) and exits, rather than a stack trace.

# Running the code

The source URL and query can be overridden on the command line (`--source-url`, `--query`) or via the environment variables above, falling back to the defaults. Pass `--help` to any script to see its options. The first run may print a one-time `langchain-community` deprecation notice, which is harmless.


# Evaluation

`eval_pipeline.py` runs the same query through the RAG chain and grades the answer three complementary ways. Each lens catches a different class of issue:

1. **Code-based.** Deterministic, free, and instant. Checks length bounds, presence of required keywords, and structure (e.g. the "three sentences maximum" instruction). Good at catching format/structure regressions, blind to meaning.
2. **Model-as-judge.** A separate LLM call scores **relevance** (does it answer the question?) and **faithfulness** (is it supported by the retrieved context, with no fabrication?). Catches semantic problems the code checks can't, at the cost of an extra API call and some non-determinism.
3. **Human-grade.** Prints the answer and asks the operator for a 1–5 rating. This is the ground truth the other two only approximate; it auto-skips when run non-interactively or when `SKIP_HUMAN_EVAL` is set.

Run it after the index pipeline:

```
python3 eval_pipeline.py
```

Output is a side-by-side JSON blob so you can see the three grades together:

```json
{
  "query": "What are the approaches to Task Decomposition?",
  "answer": "Task decomposition approaches include ... CoT ... Tree of Thoughts ...",
  "code_based": {
    "score_1_to_5": 3.8,
    "checks": {
      "non_empty": true,
      "within_length_bounds": true,
      "mentions_keywords": true,
      "at_most_three_sentences": false
    }
  },
  "model_as_judge": {
    "relevance": 5,
    "faithfulness": 5,
    "reasoning": "The answer lists the same approaches described in the context ..."
  },
  "human": {
    "score_1_to_5": null,
    "note": "skipped"
  }
}
```

Note how the lenses disagree: the answer is relevant and faithful (judge 5/5) but runs longer than the three-sentence instruction, so the code-based score drops — exactly the kind of divergence that makes multiple lenses worthwhile. To extend the eval with your own checks, add a function returning a score and wire it into the `results` dict in `main()`.

# What I would change for production

This is a teaching demo. A production deployment would add:

- **Observability.** Wire LangSmith (or another tracing backend) so each retrieval and generation is a span you can inspect. Tradeoff: a small amount of latency and a dependency on a tracing service, in exchange for being able to debug bad answers after the fact instead of guessing. Path forward: set the `LANGSMITH_*` environment variables and tag runs by query.
- **Cost guards.** Add per-user rate limiting, a token budget cap per request, and dedupe of embedding calls for unchanged source documents. Tradeoff: extra bookkeeping and a cache to maintain, against unbounded spend when traffic or document size spikes. Path forward: hash document chunks and skip re-embedding unchanged ones; reject requests over a configured token ceiling.
- **Async indexing.** Move indexing to a background worker (e.g. a task queue) instead of running it inline. Tradeoff: more moving parts (a broker, a worker) for the ability to index large corpora without blocking the request path. Path forward: enqueue an "index this URL" job and report progress out of band.
- **Citations.** Return which retrieved chunks contributed to each answer so readers can verify claims. Tradeoff: a slightly more complex prompt/response contract for trust and debuggability. Path forward: carry chunk IDs/metadata through the chain and render them alongside the answer.
- **Tests.** Add unit tests for chain composition (prompt formatting, retriever wiring) and a smoke test that runs the full pipeline against a tiny fixed corpus. Tradeoff: tests to maintain, against catching breakage from dependency bumps before users do. Path forward: mock the LLM for unit tests; gate the smoke test behind a key-present check in CI.
- **Schema versioning.** Plan for the vectorstore evolving as the source corpus, chunking strategy, or embedding model changes. Tradeoff: a migration/versioning discipline, against silently mixing incompatible embeddings in one collection. Path forward: version the collection name (e.g. include the embedding model and chunk size) and re-index on change.
